# AGENTS.md

このプロジェクトでは、実装を必要以上に抽象化しない。
短く、直接的で、既存コードに近い実装を優先する。

## 作業手順

- 変更作業を始める前に、新しいブランチを切る。ただし関連する作業は同じブランチで行う。
- ブランチは Git Flow スタイルで管理し、新しいブランチは `develop` ブランチから作る。
- 既存の未コミット変更がある場合は、ユーザーの作業として扱い、勝手に戻さない。

## 依存関係の管理

- Python ライブラリの管理は `uv` で行う。
- ライブラリの追加、更新、削除が必要な場合は、実行前に必ずユーザーへ確認する。
- `pyproject.toml` や `uv.lock` は、手作業で整合性を崩さず、`uv` の結果に合わせる。
- 一時的な都合で `pip` など別の手段を使わない。

## 実装方針

- まず既存コードの素朴な書き方に合わせる。
- 追加機能のために新しい層やヘルパを増やす前に、その場に直接書けないか考える。
- 関数量、コード量、スクリプト数はできるだけ少なくする。
- 小さくシンプルでありながら、実際の用途に耐える汎用性を持つコードにする。
- 将来の拡張を見越した抽象化は、今のコードを明確に簡単にしないなら入れない。
- 「名前を付けると意味が出る」という理由だけで 1 行ラッパーを作らない。

## 関数化の基準

- 関数化は原則として、同じ処理が 3 回以上繰り返される場合に行う。
- 1 回か 2 回しか使わない処理は、薄いラッパーにせず、呼び出し元に直接書く。
- 次のような場合は、3 回未満の利用でも関数として切り出してよい。
  - 再帰処理や状態更新など、呼び出し元に置くと明らかに読みづらい処理。
  - UI 描画、集計、パースなど、まとまった実処理を隔離することで読みやすくなる処理。
  - 仕様上の重要な変換を 1 か所に閉じ込める必要がある処理。
- ただし、この場合でも関数名と処理量が釣り合っていること。

## コメントと言語

- コメント、補足説明、作業メモなどは日本語で記入する。
- コードの識別子は、既存コードの命名規則に合わせる。
- コメントは、処理の意図や注意点が読み取りにくい場合にだけ書く。
- コードを読めば分かる内容を、日本語コメントで繰り返さない。

## 避けるもの

- `return helper(value, flag=True)` だけのような 1 行ラッパー。
- 実態より大きな名前の関数。例: 単に 1 つのキーを見るだけの `detect_format`。
- 過度な `isinstance` や事前バリデーション。外部入力の境界では守るが、内部で同じ条件を何度も確認しない。
- 使われていない形式や将来用の分岐を残すこと。
- テストのためだけに薄い公開 API を残すこと。
- 用途が重なる小さなスクリプトを増やすこと。
- 後方互換のためのコードを残すこと。

## 入力形式とデータ変換

- 対応する入力形式は、実際に必要なものに絞る。
- 使われていない形式や互換性のためだけの分岐は、明確に必要になるまで追加しない。
- 対応外の入力は、シンプルに unsupported として扱う。
- 複雑なデータ変換や traversal は、既存のパーサ、標準 API、または呼び出し元の素直な処理で済ませる。
- 表示や集計のためだけに、独自の schema 解釈や専用レイヤーを増やさない。
- プログラム内にデフォルト値は残さない。外部ファイルに配置する。

## 迷ったとき

短く、直接的で、既存コードに近い実装を選ぶ。
抽象化は「今の重複や複雑さを減らす」ために使い、「将来便利そう」だけでは使わない。
